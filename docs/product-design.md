# GateMind 产品设计

## 1. 产品定位

GateMind 是一个面向企业内部 AI/Agent 的 **MCP 网关与工具治理平台**。

它的核心目标不是“让 AI 连更多系统”，而是让 AI 在企业环境中：

- 能安全调用工具
- 不能越权访问数据
- 不能绕过审批执行高风险动作
- 不能无痕修改生产数据
- 能被完整审计、回放、追责、回滚

一句话定义：

> GateMind 让 AI 进入企业工作流，但必须在权限、策略、审批、审计的约束下工作。

## 2. 产品要解决的问题

企业一旦让 AI 接入 ERP、CRM、WMS、Shopify、飞书、项目系统、素材系统，就会立即遇到几个问题：

1. AI 到底能调用哪些工具？
2. 谁能让 AI 调这些工具？
3. 哪些工具只读，哪些能写？
4. 哪些写操作需要确认，哪些需要审批？
5. AI 拿到的数据是否需要脱敏？
6. 调用失败或误操作后，怎么追踪、回放、回滚？
7. 工具太多时，怎么只给 AI 暴露当前任务需要的能力？

所以 GateMind 不是单纯的技术网关，而是一个：

- AI 工具治理中心
- 企业能力接入中心
- 风险控制中心
- 审批执行中心
- 审计追踪中心

## 3. 产品目标

### 3.1 第一阶段目标

建立一个可落地的企业 MCP 治理底座，让企业可以先从低风险场景开始使用 AI。

第一阶段重点：

- 统一接入 AI/Agent 的工具调用
- 建立工具目录
- 建立风险分级与权限控制
- 建立确认 / 审批 / 执行分离机制
- 建立审计日志
- 支持少量业务域工具先跑通

### 3.2 不在第一阶段做的事

- 不做复杂 BI 平台
- 不做全量企业系统接入
- 不做成熟的策略 DSL
- 不做复杂组织架构引擎
- 不做完整运营后台大而全功能

## 4. 目标用户

### 4.1 平台管理员

职责：

- 配置系统基础参数
- 管理工具注册与发布
- 管理风险等级、审批模板、策略规则
- 查看平台整体运行情况

### 4.2 业务负责人

职责：

- 审批高风险变更
- 查看业务域内工具使用情况
- 审核业务域工具上线范围

### 4.3 普通业务用户

职责：

- 通过 AI 发起查询、草稿生成、轻量操作
- 对 Level 2 操作做二次确认
- 查看自己的操作记录

### 4.4 审计 / 风控角色

职责：

- 查询审计日志
- 追踪关键操作链路
- 回溯谁发起、谁审批、执行前后发生了什么

### 4.5 Agent / AI 客户端

职责：

- 通过统一入口请求工具
- 根据当前身份和上下文动态获取可用工具
- 按策略提示执行 preview、confirm、approval 流程

注意：

Agent 是“系统调用方”，不是后台运营用户。

## 5. 核心价值

GateMind 的核心价值不是“多接几个系统”，而是同时解决五件事：

1. 统一入口：所有 AI 调用都收口到一个网关。
2. 统一治理：工具注册、风险分级、权限控制、审批规则统一管理。
3. 统一执行：不同业务域能力通过稳定工具语义暴露，而不是散乱接口。
4. 统一审计：所有调用、审批、执行都有日志和可追踪链路。
5. 统一扩展：后续能逐步接更多业务域和更多企业系统。

## 6. 产品边界

### 6.1 产品负责什么

- 管理工具目录
- 动态暴露工具
- 做权限校验和风险判断
- 发起确认和审批
- 记录审计
- 路由到对应业务域 Provider

### 6.2 产品不直接负责什么

- 不直接替代 ERP/WMS/CRM 等业务系统
- 不负责业务系统原始权限体系建设
- 不负责 AI 大模型本身能力
- 不负责业务系统最终业务规则定义

GateMind 更像“AI 调企业能力的守门层”。

## 7. 产品能力地图

产品能力分成六层：

### 7.1 第一层：统一接入层

能力：

- 统一 API / MCP Gateway 入口
- Agent 鉴权
- 请求追踪
- 会话上下文装配

### 7.2 第二层：工具目录层

能力：

- 工具注册
- 工具版本管理
- 工具元数据维护
- 工具上下线
- 工具所属业务域和风险等级管理

### 7.3 第三层：策略与权限层

能力：

- 根据用户、Agent、会话、环境、业务域决定是否允许调用
- 决定是直接执行、要求确认、要求审批还是拒绝
- 注入脱敏与字段过滤规则

### 7.4 第四层：变更与审批层

能力：

- 预览高风险变更
- 创建变更单
- 分配审批人
- 审批通过后执行
- 审批拒绝或过期关闭

### 7.5 第五层：执行与回滚层

能力：

- 路由到业务域 Provider
- 调用真实业务系统
- 保存执行记录
- 保存必要的回滚信息

### 7.6 第六层：审计与追踪层

能力：

- 查询调用历史
- 查看审批历史
- 查看策略命中结果
- 查看输入输出快照
- 回放关键链路

## 9. 风险等级设计

### 9.1 Level 0

只读查询。

例子：

- 查询库存
- 查询项目状态
- 查询素材详情

要求：

- 自动执行
- 权限校验
- 结果脱敏
- 记录审计

### 9.2 Level 1

草稿生成，不直接写生产。

例子：

- 生成商品文案草稿
- 生成周报草稿
- 生成标签建议

要求：

- 自动执行
- 记录审计
- 不直接写业务系统生产数据

### 9.3 Level 2

轻量写入，需要用户确认。

例子：

- 更新标签
- 创建普通任务
- 修改非关键字段

要求：

- 先预览
- 要求用户确认
- 执行后记录审计

### 9.4 Level 3

高风险变更，需要负责人审批。

例子：

- 发布商品
- 调整项目排期
- 同步生产数据

要求：

- 强制 preview
- 创建变更单
- 审批通过再执行
- 执行前再次校验

### 9.5 Level 4

极高风险，涉及钱、库存、权限、删除、批量操作。

例子：

- 批量改价
- 批量改库存
- 删除数据
- 改权限

要求：

- 多人审批
- 默认禁止 AI 自动直写
- 必须有回滚能力或恢复方案

## 10. 核心流程设计

### 10.1 查询类流程

```text
用户发起任务
 -> Agent 请求工具
 -> GateMind 动态筛出可用工具
 -> 策略校验
 -> 调用业务域 Provider
 -> 返回结果
 -> 审计记录
```

### 10.2 草稿类流程

```text
用户发起任务
 -> Agent 请求草稿类工具
 -> 策略放行
 -> 生成草稿
 -> 返回草稿
 -> 审计记录
```

### 10.3 轻量写入流程

```text
用户发起任务
 -> Agent 请求写入工具
 -> 策略判断为 confirm
 -> 返回预览结果
 -> 用户确认
 -> 执行
 -> 审计记录
```

### 10.4 高风险变更流程

```text
用户发起任务
 -> Agent 请求高风险工具
 -> 策略判断为 approval_required
 -> 生成预览
 -> 创建变更单
 -> 审批
 -> 审批通过后再次校验
 -> 执行
 -> 记录审计
 -> 必要时回滚
```

## 11. MVP 范围

第一阶段只做真正有价值、且风险可控的能力。

### 11.1 MVP 模块

- 统一 Gateway 调用入口
- Tool Registry
- Policy Engine
- Approval Center
- Audit Center
- Provider Router

### 11.2 MVP 工具类型

- Level 0 查询工具
- Level 1 草稿工具
- 少量 Level 2/3 演示工具

### 11.3 MVP 业务域建议

优先选最容易体现价值、又相对低风险的业务域：

- 项目域
- 素材域
- 商品域中的查询/草稿能力

### 11.4 MVP 示例工具

- `search_tools`
- `get_tool_detail`
- `get_project_status`

- `generate_weekly_report_draft`
- `generate_product_copy_draft`
- `update_tag_preview`
- `create_change_request`
- `approve_change_request`
- `execute_change_request`

## 12. 后台页面设计

第一阶段后台不求花哨，重点要能支撑治理闭环。

### 12.1 控制台首页

目标：

- 看平台整体运行情况
- 看调用量、审批量、拒绝量、异常量
- 看高风险操作趋势

核心模块：

- 今日调用概览
- 风险等级分布
- 待审批变更单
- 最近异常调用
- 热门工具排行

### 12.2 工具目录页

目标：

- 维护工具定义和版本
- 查看工具所属业务域与风险等级

核心功能：

- 工具列表
- 工具搜索筛选
- 工具详情
- 工具注册 / 编辑 / 上下线
- 查看输入输出 schema

### 12.3 策略规则页

目标：

- 配置谁在什么场景可以调什么工具
- 配置脱敏、限流、审批要求

核心功能：

- 策略列表
- 策略优先级
- 角色和业务域限制
- 风险等级限制
- 审批模板绑定

### 12.4 变更审批页

目标：

- 处理所有待审批高风险动作

核心功能：

- 待审批列表
- 变更单详情
- 预览变更内容
- 审批通过 / 驳回
- 查看执行与回滚状态

### 12.5 审计日志页

目标：

- 查询调用历史与责任链路

核心功能：

- 按工具/用户/业务域/时间筛选
- 查看请求和响应快照
- 查看命中策略
- 查看审批记录
- 查看 trace 详情

### 12.6 业务域管理页

目标：

- 管理业务域 Provider 与工具归属

核心功能：

- 业务域列表
- 域内工具查看
- Provider 连接状态
- 域负责人配置

## 13. 关键交互原则

### 13.1 永远先展示风险信息

对于写入类操作，必须先给出：

- 风险等级
- 影响范围
- 是否生产
- 是否可回滚
- 是否需要审批

### 13.2 审批人看的是业务动作，不是技术参数

审批页不能让审批人看一堆 JSON 参数，应该优先展示：

- 将修改什么
- 影响多少对象
- 谁发起的
- 为什么发起
- 风险等级
- 回滚方式

### 13.3 审计信息默认可追溯

所有关键动作都要自动带：

- 发起人
- Agent
- 时间
- 工具版本
- 策略结果

### 13.4 AI 只能看到当前任务必要工具

不要把全量工具暴露给 Agent。

应根据：

- 当前用户身份
- 当前业务域
- 当前任务类型
- 当前风险等级

动态缩小工具集合。

## 14. 权限模型建议

第一阶段采用简单但清晰的 RBAC。

角色建议：

- `platform_admin`
- `domain_owner`
- `approver`
- `auditor`
- `operator`
- `agent_client`

控制粒度：

- 页面访问权限
- 工具查看权限
- 工具调用权限
- 审批权限
- 审计查看权限
- 策略配置权限

后续再演进到更细的 ABAC。

## 15. 成功指标

第一阶段关注是否形成治理闭环，而不是工具数量。

关键指标：

- 有多少工具完成注册治理
- AI 调用通过网关的占比
- 高风险动作审批覆盖率
- 审计日志完整率
- 查询类工具调用成功率
- 草稿类工具使用次数
- 高风险误执行事件数

## 17. 当前产品结论

GateMind 第一阶段最合理的产品形态不是“大而全企业平台”，而是：

一个围绕 **工具目录、策略、审批、审计、执行路由** 构建的企业 AI 治理中台。

它要先解决：

- AI 能不能安全调用
- 高风险操作能不能被拦住
- 审批能不能接上
- 调用能不能追踪
- 问题发生后能不能说清楚

只要这五件事成立，后续接更多系统和业务域才有意义。
