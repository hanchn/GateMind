# 企业 MCP 智能网关整体设计思路

企业 MCP 的核心，不是给每个系统单独开发一个 MCP Server，然后让 Agent 随便调用。

真正企业级的做法，是在 Agent 和企业系统之间加一层 **智能网关**。

这个智能网关的作用，不是替 AI 干活，也不是简单转发 API，而是负责判断：

AI 想调用什么能力？
这个能力能不能调用？
谁能调用？
一次能查多少？
调用频率是多少？
是否涉及敏感数据？
是否涉及生产变更？
是否需要审批？
执行后能不能追踪和回滚？

所以企业 MCP 的整体架构应该是：

```text
Agent / AI 客户端
  ↓
企业 MCP 智能网关
  ↓
工具目录 / 权限策略 / 调用限制 / SQL Guard / 审批流 / 审计日志
  ↓
业务域 MCP Server
  ↓
企业应用 / 数据库 / API / 工作流
```

智能网关是企业 Agent 的总闸门。

所有 MCP Tool、SQL、API、工作流调用，都必须先经过它。

---

## 一、第一层：接入控制

企业 MCP 首先要解决的是：谁能进来。

所以必须有白名单或者 Token。

不是任何 Agent、任何 MCP Server、任何 SQL 工具都能直接接入企业系统。

所有接入方都必须登记：

```text
Agent 是谁
MCP Server 是谁
所属系统是什么
负责人是谁
允许调用哪些工具
属于测试环境还是生产环境
Token 是否有效
权限范围是什么
```

可以分成几类白名单：

```text
Agent 白名单
MCP Server 白名单
业务系统白名单
工具白名单
IP 白名单
环境白名单
```

同时，每次调用都要携带 Token。

Token 里最好能识别：

```text
应用身份
用户身份
Agent 身份
权限范围
过期时间
环境信息
```

不能用一个全局万能 Token。

白名单和 Token 解决的是：

```text
谁可以接入？
谁可以调用？
调用的是不是合法来源？
```

---

## 二、第二层：默认调用限制

光有白名单不够。

因为就算是合法 Agent，也可能因为模型循环、误调用、参数错误，把数据库或者业务系统打爆。

所以平台必须有默认调用限制。

至少包括：

```text
单次调用条数限制
调用频率限制
返回数据大小限制
执行超时限制
重复调用检测
最大分页限制
```

比如：

```text
单次默认最多返回 100 条
普通用户最多 500 条
管理员最多 1000 条
超过必须分页
超过导出阈值要审批
```

频率也要限制：

```text
普通查询工具：每分钟 60 次
SQL 查询工具：每分钟 10 次
库存查询工具：每分钟 30 次
高风险工具：每小时 5 次
审批类工具：不允许自动循环调用
```

还要防止 Agent Loop。

比如同一个用户、同一个工具、同一组参数，短时间内重复调用超过 3 次，就要阻断或者降级处理。

这层解决的是：

```text
能不能无限查？
能不能批量导出？
能不能疯狂循环调用？
会不会把系统打爆？
```

---

## 三、第三层：工具注册表

企业应用很多，不能靠人临时判断哪个工具危险。

每个 MCP Tool 上架之前，都要进入工具注册表。

工具注册时要写清楚：

```text
工具名称
所属系统
所属业务域
负责人
工具描述
输入参数
输出字段
是否只读
是否写入
是否批量
是否影响生产
是否涉及钱
是否涉及库存
是否涉及客户数据
是否涉及权限
是否可回滚
风险等级
审批规则
调用限制
日志级别
工具版本
```

比如：

```text
工具：query_inventory_summary
系统：库存系统
类型：只读查询
风险等级：Level 0
审批：不需要
限制：最多返回 500 行
数据要求：不返回供应商成本价
```

再比如：

```text
工具：update_product_price
系统：Shopify
类型：生产写入
风险等级：Level 4
审批：运营负责人 + 财务负责人
限制：必须先生成变更单
回滚：必须保存原价格快照
```

工具注册表解决的是：

```text
这个工具是什么？
它属于什么风险？
谁负责？
什么情况下能调用？
什么情况下要审批？
```

---

## 四、第四层：风险分级

企业 MCP 不能所有操作都审批。

如果所有东西都审批，AI 就没效率。

所以要按风险分级。

可以分成五级：

```text
Level 0：只读查询
自动执行，但要做权限校验、数据脱敏和日志记录。

Level 1：生成草稿
可以自动执行，但不写生产，比如生成周报草稿、商品描述草稿、SEO 草稿。

Level 2：轻量写入
需要用户二次确认，比如修改标签、创建普通任务、保存草稿。

Level 3：高风险变更
需要负责人审批，比如发布商品、同步生产、调整项目排期。

Level 4：钱、库存、权限、删除、批量操作
需要多人审批，甚至禁止 AI 直接执行。
```

比如素材中台：

```text
查素材：Level 0
生成素材周报：Level 1
修改素材标签：Level 2
发布素材：Level 3
删除素材、修改权限：Level 4
```

Shopify：

```text
查商品：Level 0
生成商品描述：Level 1
更新 SEO 字段：Level 2
发布商品：Level 3
改价格、改库存、删除商品：Level 4
```

风险分级解决的是：

```text
哪些能自动执行？
哪些要确认？
哪些要审批？
哪些要禁止？
```

---

## 五、第五层：SQL Guard

如果别人要接 SQL，不能直接暴露给 Agent。

尤其不能暴露这种万能工具：

```text
execute_sql(sql)
```

这是非常危险的。

正确做法是：

```text
Agent
  ↓
智能网关
  ↓
SQL Guard
  ↓
只读库 / 分析库 / 业务 API
```

SQL Guard 要做这些事情：

```text
SQL 解析
只允许 SELECT
禁止 UPDATE / DELETE / INSERT / DROP / ALTER
禁止访问敏感表
禁止访问敏感字段
必须带 LIMIT
限制扫描行数
限制执行时间
结果自动脱敏
所有查询记录日志
```

只读 SQL 可以在限制下执行。

敏感 SQL 要脱敏、聚合，必要时审批。

写入 SQL 默认禁止 AI 直接执行。

如果 AI 想改价格、改库存、改权限，不应该让它写 SQL，而是转成固定业务工具：

```text
preview_price_change
create_price_change_request
execute_approved_price_change
```

SQL Guard 解决的是：

```text
AI 能不能查数据库？
能查哪些表？
能查哪些字段？
能查多少？
能不能写生产库？
```

---

## 六、第六层：审批流

审批流不是审批 SQL，而是审批业务动作。

AI 准备执行高风险操作时，智能网关应该把它变成变更单。

比如 AI 想批量改价，不应该让审批人看：

```sql
UPDATE product SET price = price * 0.8 WHERE category = 'sun_hat';
```

而应该让审批人看：

```text
变更类型：批量修改商品价格
系统：Shopify
对象：32 个 SKU
修改前：平均价格 29.99 美元
修改后：平均价格 23.99 美元
降幅：20%
影响范围：美国站商品售价、毛利、广告 ROI
AI 理由：近 14 天销量下降，库存周转超过 60 天
风险等级：Level 4
审批人：运营负责人 + 财务负责人
回滚方式：恢复原价格快照
```

审批流程应该是：

```text
Agent 生成操作计划
  ↓
智能网关判断风险
  ↓
生成变更单
  ↓
负责人审批
  ↓
审批通过后再次校验数据
  ↓
执行固定业务工具
  ↓
记录审计日志
  ↓
必要时支持回滚
```

审批流解决的是：

```text
高风险操作谁批准？
批准的到底是什么？
修改前后是什么？
影响范围是什么？
出错能不能回滚？
```

---

## 七、第七层：数据脱敏

企业 MCP 不只是工具有风险，数据本身也有风险。

比如订单数据里可能有：

```text
客户姓名
手机号
邮箱
地址
支付信息
内部备注
```

这些不能默认返回给 AI。

数据要分级：

```text
公开数据
内部普通数据
内部敏感数据
客户隐私数据
财务数据
权限数据
密钥数据
生产配置
```

对应策略是：

```text
普通数据可以返回
敏感数据脱敏返回
客户数据按权限最小化返回
财务数据优先返回汇总
密钥数据永远不能返回
生产配置默认只读
```

比如查订单，AI 默认只应该看到：

```text
订单号
订单状态
商品信息
物流状态
异常原因
```

不应该默认看到：

```text
手机号
详细地址
支付信息
客户隐私
```

数据脱敏解决的是：

```text
AI 能看到什么？
哪些字段要隐藏？
哪些字段要脱敏？
哪些数据只能返回汇总？
```

---

## 八、第八层：动态工具暴露

企业里如果有几百个工具，不能一次性全部暴露给模型。

工具越多，Agent 越容易选错，也越危险。

所以智能网关要根据当前场景动态暴露工具。

比如用户问素材问题，只暴露：

```text
search_materials
get_material_detail
get_project_assets
get_material_usage_stats
generate_asset_report
```

不要同时暴露：

```text
change_product_price
delete_order
modify_permission
sync_to_prod
execute_sql
```

动态工具暴露可以按这些维度判断：

```text
用户身份
Agent 类型
当前任务
业务域
风险等级
数据权限
当前环境
```

这层解决的是：

```text
这次对话里，AI 应该看到哪些工具？
哪些工具根本不该出现在上下文里？
```

---

## 九、第九层：审计日志

企业 MCP 最后一定要能追踪。

每次调用都要记录：

```text
用户是谁
Agent 是谁
会话是什么
调用了哪个工具
传了什么参数
返回了什么结果
命中了什么策略
有没有脱敏
有没有审批
谁审批的
执行前是什么
执行后是什么
是否成功
是否可回滚
工具版本是多少
trace_id 是什么
```

因为出了问题后，企业一定会问：

```text
是谁让 AI 做的？
AI 为什么这么做？
调用了哪些系统？
改了哪些数据？
谁审批的？
能不能恢复？
```

没有审计日志，企业 MCP 就不能大规模上线。

审计日志解决的是：

```text
出事后能不能追？
能不能解释？
能不能回滚？
能不能追责？
```

---

## 十、整体落地顺序

企业 MCP 不要一上来就做全自动变更。

最稳的落地顺序是三步。

第一步，先做只读能力：

```text
查询
统计
汇总
异常分析
生成报告
```

比如查素材、查库存、查项目进度、生成周报。

这一层风险最低，价值最明显。

第二步，做草稿能力：

```text
生成商品描述草稿
生成 SEO 草稿
生成任务草稿
生成素材标签建议
生成周报草稿
生成活动页草稿
```

这一层能提效，但不直接改生产。

第三步，再做变更能力：

```text
修改标签
更新 SEO
创建任务
调整排期
发布商品
修改价格
修改库存
```

这一层必须接入确认、审批、审计和回滚。

---

## 最终一句话

企业 MCP 智能网关的核心，不是让 AI 直接操作企业系统，而是：

```text
让 AI 在白名单、Token、调用限制、权限控制、数据脱敏、风险分级、审批流和审计日志的约束下，安全调用企业能力。
```

它不是普通 API 网关。

它更像：

```text
API Gateway
+ Tool Registry
+ Policy Engine
+ SQL Guard
+ Approval Center
+ Audit Center
+ Data Masking
+ Agent Tool Router
```

它的价值是：

```text
让 AI 能进入企业工作流，但不能越权、不能乱查、不能乱改、不能绕过审批、不能无痕操作。
```

所以这个项目可以定义为：

```text
企业级 MCP 智能网关
```

或者用之前的项目名：

```text
GateMind：企业级 Agent 工具治理与 MCP 网关平台
```
